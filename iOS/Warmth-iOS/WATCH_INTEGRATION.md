# Warmth — Apple Watch companion + complication: integration spec

This document tells the parent agent exactly how to wire the **watchOS app** and
**WidgetKit complication** into the shared XcodeGen project **after** the iOS
worker has finished. Nothing here was applied automatically — `project.yml`, the
generated `.pbxproj`, and the iOS worker's files were intentionally left untouched.

All paths are relative to the XcodeGen root: `iOS/Warmth-iOS/` (the folder that
contains `project.yml` and `Warmth.xcodeproj`). Source files live one level down,
under `Warmth-iOS/…` (note the doubled folder name in this repo).

---

## 1. Files that were created (new, disjoint folders only)

```
Warmth-iOS/WarmthWatch/                         # watchOS app target sources
  App/WarmthWatchApp.swift                      # @main App + RootWatchView (idle/recording switch)
  DesignSystem/WatchTheme.swift                 # ember palette, Space Grotesk fonts, PulsingEmberIndicator
  Views/RecordingStateView.swift                # live hero: pulse + timer + Stop + last person/org
  Views/StartView.swift                         # idle "Start capturing" screen + PhoneUnreachableHint
  Services/WatchConnectivityService.swift       # @Observable WC client (state + send toggle/start/stop)
  Shared/WatchSharedState.swift                 # App-Group snapshot shared with the complication
  Resources/Fonts/SpaceGrotesk-{Regular,Medium,SemiBold,Bold}.ttf

Warmth-iOS/WarmthWatchWidget/                   # widget extension target sources
  WarmthWatchWidget.swift                        # @main WidgetBundle + Widget (supportedFamilies)
  ComplicationProvider.swift                     # TimelineProvider (reads App-Group snapshot)
  ComplicationViews.swift                        # per-family layouts (circular/corner/inline/rectangular)

Warmth-iOS/Warmth/Services/Connectivity/
  WatchSessionService.swift                      # iOS-side @Observable WC bridge (+ MockWatchSessionService)
```

> `WatchSharedState.swift` is compiled into **both** the watch app and the widget
> (see the widget target `sources` below).

## 2. `project.yml` additions

Add the two targets below to the `targets:` map. Keep the existing `Warmth` and
`WarmthTests` targets as-is, except for the one embed dependency noted in 2.3.

### 2.1 watchOS app target

```yaml
  WarmthWatch:
    type: application
    platform: watchOS
    deploymentTarget: "26.0"
    sources:
      - path: Warmth-iOS/WarmthWatch
        excludes:
          - "**/.DS_Store"
          - "Info.plist"                   # generated by the info: block below
    dependencies:
      - target: WarmthWatchWidget          # watch app embeds the widget extension
        embed: true
    settings:
      base:
        PRODUCT_BUNDLE_IDENTIFIER: com.warmth.gtmhackathon.watchkitapp
        PRODUCT_NAME: Warmth
        TARGETED_DEVICE_FAMILY: "4"        # 4 = Apple Watch
        ENABLE_PREVIEWS: YES
        GENERATE_INFOPLIST_FILE: NO
        WATCHOS_DEPLOYMENT_TARGET: "26.0"
        # App Group lets the complication read live capture state (see §4).
        CODE_SIGN_ENTITLEMENTS: Warmth-iOS/WarmthWatch/WarmthWatch.entitlements
    info:
      path: Warmth-iOS/WarmthWatch/Info.plist
      properties:
        CFBundleDisplayName: Warmth
        CFBundleShortVersionString: "$(MARKETING_VERSION)"
        CFBundleVersion: "$(CURRENT_PROJECT_VERSION)"
        WKApplication: true                # modern single-target watch app (no extension)
        WKCompanionAppBundleIdentifier: com.warmth.gtmhackathon
        UIAppFonts:
          - SpaceGrotesk-Regular.ttf
          - SpaceGrotesk-Medium.ttf
          - SpaceGrotesk-SemiBold.ttf
          - SpaceGrotesk-Bold.ttf
```

### 2.2 watchOS widget (complication) extension target

```yaml
  WarmthWatchWidget:
    type: app-extension
    platform: watchOS
    deploymentTarget: "26.0"
    sources:
      - path: Warmth-iOS/WarmthWatchWidget
        excludes:
          - "**/.DS_Store"
          - "Info.plist"                   # generated by the info: block below
      # Shared snapshot model compiled into the widget too:
      - path: Warmth-iOS/WarmthWatch/Shared/WatchSharedState.swift
    settings:
      base:
        PRODUCT_BUNDLE_IDENTIFIER: com.warmth.gtmhackathon.watchkitapp.complication
        PRODUCT_NAME: WarmthWatchWidget
        TARGETED_DEVICE_FAMILY: "4"
        ENABLE_PREVIEWS: YES
        GENERATE_INFOPLIST_FILE: NO
        WATCHOS_DEPLOYMENT_TARGET: "26.0"
        CODE_SIGN_ENTITLEMENTS: Warmth-iOS/WarmthWatchWidget/WarmthWatchWidget.entitlements
    info:
      path: Warmth-iOS/WarmthWatchWidget/Info.plist
      properties:
        CFBundleDisplayName: Warmth
        CFBundleShortVersionString: "$(MARKETING_VERSION)"
        CFBundleVersion: "$(CURRENT_PROJECT_VERSION)"
        NSExtension:
          NSExtensionPointIdentifier: com.apple.widgetkit-extension
```

> WidgetKit widgets use a `@main WidgetBundle`, so **no** `NSExtensionPrincipalClass`
> is required — only `NSExtensionPointIdentifier: com.apple.widgetkit-extension`.

### 2.3 iOS app embeds the watch app

Add one dependency to the existing `Warmth` target so the phone build bundles the
watch app (which in turn embeds the widget):

```yaml
  Warmth:
    ...
    dependencies:
      - package: Firebase
        product: FirebaseAuth
      # ... existing deps ...
      - target: WarmthWatch                # iOS app embeds the watch app
        embed: true
```

> Embedding chain: **Warmth (iOS) → WarmthWatch (watch app) → WarmthWatchWidget (widget)**.
> XcodeGen creates the correct "Embed Watch Content" / "Embed App Extensions" build
> phases from the `embed: true` flags above.

### 2.4 Scheme for the watch app

```yaml
schemes:
  # ... existing Warmth scheme ...
  WarmthWatch:
    build:
      targets:
        WarmthWatch: all
    run:
      config: Debug
```

## 3. Entitlements (required for the live complication)

The complication reads capture state from a shared **App Group**. Create both
entitlement files (App Group enables cross-process `UserDefaults`):

`Warmth-iOS/WarmthWatch/WarmthWatch.entitlements`
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>com.apple.security.application-groups</key>
  <array>
    <string>group.com.warmth.gtmhackathon</string>
  </array>
</dict>
</plist>
```

`Warmth-iOS/WarmthWatchWidget/WarmthWatchWidget.entitlements` — identical contents.

> The App Group id must match `WatchSharedStore.appGroupID` in
> `Warmth-iOS/WarmthWatch/Shared/WatchSharedState.swift`
> (`group.com.warmth.gtmhackathon`).
>
> **Graceful fallback:** if the App Group is not provisioned, `WatchSharedStore`
> falls back to `UserDefaults.standard`; the watch app still works fully and the
> complication simply shows idle branding instead of live status. So the App Group
> is required only for the *live* complication, not for a passing build.

## 4. The single iOS hook

`WatchSessionService` is self-contained (no dependency on the capture/speech
stack). Wire it in **one place** where the app owns both the WC bridge and the
capture service (e.g. `AppModel`, the root view model, or `RootView`):

```swift
// 1. Own a single instance (e.g. on AppModel / your @Observable root state)
let watch = WatchSessionService()

// 2. Wrist → phone: connect the two intent closures to your capture service.
//    (capture service may be SpeechService / CaptureViewModel / RecordingEngine —
//     only "start" and "stop" entry points are needed.)
watch.onStartRequested = { [weak captureService] in captureService?.startCapture() }
watch.onStopRequested  = { [weak captureService] in captureService?.stopCapture() }

// 3. Phone → watch: every time recording state (or the latest detected person)
//    changes, mirror it to the watch + complication:
watch.updateState(
    isRecording:    captureService.isRecording,
    elapsed:        captureService.elapsedSeconds,           // TimeInterval since start; 0 if idle
    lastPersonName: captureService.lastDetectedPerson?.name, // optional
    lastPersonOrg:  captureService.lastDetectedPerson?.company
)
```

Call `updateState(...)` from wherever capture state changes are observed — e.g. a
`didSet` on the capture service's `isRecording`, a Combine sink, or `.onChange`
in the root view. That single call keeps both the watch UI and the complication
correct.

If the iOS worker exposes capture as an `@Observable`, the cleanest wiring is:

```swift
.onChange(of: captureService.isRecording) { _, _ in
    watch.updateState(isRecording: captureService.isRecording,
                      elapsed: captureService.elapsedSeconds,
                      lastPersonName: captureService.lastDetectedPerson?.name,
                      lastPersonOrg: captureService.lastDetectedPerson?.company)
}
```

No other iOS files need to change.

## 5. WatchConnectivity protocol (both directions)

- **Watch → phone** (`sendMessage` when reachable, else `updateApplicationContext`):
  `["action": "toggleRecording" | "startRecording" | "stopRecording"]`
- **Phone → watch** (`updateApplicationContext` for latest-state sync, plus
  `sendMessage` when reachable for instant delivery):
  ```
  ["action": "recordingStateChanged",
   "isRecording": Bool,
   "elapsed": TimeInterval,
   "lastPersonName": String?,   // omitted when nil
   "lastPersonOrg": String?]    // omitted when nil
  ```
- Both sides handle `activationDidCompleteWith`; the iOS side also implements the
  iOS-only `sessionDidBecomeInactive` / `sessionDidDeactivate` (re-activates).
- Resilience: when the counterpart is unreachable the watch shows a subtle
  "Phone not reachable" hint and the intent is queued via application context.

## 6. Build / verify commands (run AFTER the iOS worker finishes)

```bash
cd iOS/Warmth-iOS

# 1. Regenerate the project with the new targets
xcodegen generate

# 2. Confirm the new targets/schemes exist
xcodebuild -list -project Warmth.xcodeproj

# 3. Build the iOS app (also builds the embedded watch app + widget)
xcodebuild build -scheme Warmth \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' -quiet

# 4. Build the watch app directly
xcodebuild build -scheme WarmthWatch \
  -destination 'platform=watchOS Simulator,name=Apple Watch Series 11 (46mm)' -quiet
```

Pick available device names from `xcrun simctl list devices available` if the
defaults don't exist. For end-to-end connectivity, run a **paired** iPhone + Watch
simulator pair.

## 7. Notes / blockers

- **Final build requires the iOS worker to finish first** (shared `project.yml`
  and `.pbxproj`). This spec is purely additive; the parent should merge the
  `project.yml` blocks above, add the two `.entitlements` files, create the two
  `Info.plist` files (XcodeGen generates them from the `info:` blocks above — no
  manual plist needed), then run `xcodegen generate`.
- **App Group provisioning** (`group.com.warmth.gtmhackathon`) must be enabled on
  the Apple Developer account for the live complication. Without it the app still
  builds and runs; the complication shows idle branding only.
- **Signing**: `DEVELOPMENT_TEAM` is empty in `project.yml`. The watch + widget
  targets inherit it; a team must be selected once in Xcode for device installs
  (simulator builds work without signing).
- The widget uses system fonts (not Space Grotesk) on purpose — accessory
  complications render largely monochrome/system-styled, so this avoids bundling
  fonts into the extension. The watch **app** bundles Space Grotesk via `UIAppFonts`.
